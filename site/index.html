01 · Drift engine

Typecheck passes.
Runtime throws.

A returns validator is a promise about what your function sends back. Your agent adds the field, ships the feature — and never touches the validator. TypeScript can't see the drift. Convex enforces it at runtime.

convex-doctor walks schema → validator → every return path of the handler and diffs all three. It understands the code you actually write: foreign-key joins, enrichment spreads, .paginate() envelopes, null guards, both-branch ternaries, Promise.all maps, and validators imported across files.

  args: { id: v.id("users") },
  returns: v.object({
    name: v.string(),
    // email? never updated —
    // throws on the first real row
  }),
  handler: (ctx, { id }) => ctx.db.get(id),
})
02 · Best-practice lints

The whole guide,
enforced.

Every rule from the Convex best-practices guide and the official eslint-plugin — plus the ones they don't ship — in a single pass with no install and no config. Each diagnostic explains why it matters, shows a concrete fix, and deep-links the exact doc section it enforces — written for whoever's reading, human or agent.

Filter_in_querywarn
.filter() on a db query scans every row and discards in memory. Use .withIndex — or filter in plain TypeScript.
Await_in_loopwarn
await ctx.db.* inside a for-loop is sequential round-trips. Parallelize with Promise.all. Pagination cursors are exempt.
Unbounded_collectwarn
.collect() with no index narrowing can load the whole table. Bound it with .withIndex, .take(n), or .paginate().
Fetch_in_queryerror
The V8 isolate has no fetch — a query that calls it throws, every time. That code belongs in an action.
Db_in_actionerror
ActionCtx has no ctx.db. Reads and writes go through ctx.runQuery / ctx.runMutation.
Nondeterministic_querywarn
Date.now() inside a query freezes in the reactive cache — the value never updates with wall-clock time.
Schedule_public_fnwarn
Schedulers and crons should call internal.* — a public api.* function is reachable by any client.
Floating_ctx_promisewarn
An un-awaited ctx.* write or schedule may never run — and its errors are swallowed silently.
Missing_arg_validatorwarn
A public function with no args validator lets client input reach the handler unchecked.
Query_in_node_fileerror
A query in a "use node" file can't run in the V8 isolate — Convex rejects the deploy outright.
Duplicate_cron_iderror
Two cron jobs registered under one identifier — another deploy-time rejection caught at your desk instead.
Redundant_indexwarn
by_a is dead weight when by_a_b exists — a prefix index duplicates what the longer one already covers.

03 · Dead-function detection

Dead means unreachable,
not just unreferenced.

--dead builds a call graph across your whole project and lists every Convex function no caller reaches. A reference from a function that is itself dead grants no life — so an orphaned entry point drags its entire private-helper cluster into the dead list in one pass.

  • ✓
    api.* / internal.* chains — resolved through barrel re-exports.
  • ✓
    String function names — "path/file:fn" literals from npx convex run scripts and HTTP clients count as callers.
  • ✓
    Self-calls don't count — a paged migration re-scheduling itself is not "alive".
  • ✓
    Keep comments — document why an external entry point stays, right next to the code.

04 · Agentic usage

Built to be driven
by an agent.

An agent wrote most of this code — so hand the findings straight back to it. Don't dump 455 issues into a context window: convex-doctor works one rule code at a time. The agent locks a group, fixes every site with one shared recipe, re-scans to verify, commits, and moves on. The loop converges to zero — in clean, reviewable, per-group commits.

Step 1 · The menu

List fixable groups

One tiny line per rule code, priority-ordered: errors first, then warnings, then info.

$ convex-doctor groups --json
Step 2 · The batch

Lock one group

A bounded work-list: the shared recipe once, then just file · line · function per site.

$ convex-doctor --only AWAIT_IN_LOOP --json
Step 3 · The cursor

Re-scan to verify

Fixed sites vanish from the next run — re-scanning is the cursor. Loop until total is 0, commit the group.

$ convex-doctor agent-guide
Every group tells the agent how hard to think
Mechanical

The diagnostic fully determines the edit — remove or move a line. Safe to apply without reading around it.

Guided

A deterministic recipe, but the agent reads local context first — the schema type, the loop body, the query chain.

Manual

Architectural, cross-file, or data-migration judgment. Reason carefully; maybe ask a human.

claude> /convex-fix

Install the /convex-fix skill and your agent runs the whole loop itself — audit, fix, verify, commit.
