01 · The problem

Logs lie.
The bill doesn't.

Log streams drop events, can't see storage-API calls, and are blind to the egress your own cloud backups generate. The only truthful source is the usage API the Convex dashboard itself reads — exact billed numbers, per day, per project, per function, updating hourly.

The watchdog polls it every few hours, keeps a local history, and tells you the day something changes — which function, how many bytes per call, and what it does to this month's bill. Billing is per team, pooled across every project: one noisy side project eats the same included tiers as production. The watchdog watches all of them at once.

02 · What your agent builds

A collector, five detectors,
one dashboard.

The spec hands your agent the endpoints, query ids, column mappings, units, and pricing — it writes the collector in your stack, in a directory you own. The dashboard it doesn't improvise: it forks the reference dashboard and regenerates only its data block — so every watchdog looks like this one, not whatever the model felt like that day.

Tier_metersdash
Every included tier — function calls, DB bandwidth, egress, compute, storage — used, projected, and priced in dollars.
Attributiondash
Which project — then which function — is eating each tier. Share bars, bytes-per-call, top movers.
New_functionalert
A function never seen before is suddenly a top consumer. The #1 real-world bill spike: new code, unindexed query.
Cost_shapealert
A known function's bytes-per-call jumps ≥3× its 14-day median — the query got heavier, not more popular.
Tier_trajectoryalert
Month-to-date pace crosses an included tier: the crossing date and the projected overage, in dollars, before it happens.
Intraday_spikealert
Usage rows update hourly. Today-so-far vs a typical day catches a runaway loop the afternoon it ships — optionally straight to a Discord/Slack webhook.
03 · Local by design

Your token never
leaves your machine.

The watchdog reads the access token your npx convex login already wrote to ~/.convex/config.json, and the spec is explicit: send it to api.convex.dev and nowhere else. Read-only GET requests. Snapshots stay on disk. No server, no signup, no telemetry.

1

Copy the prompt

2

The agent builds it

It fetches the spec, proves auth against your team, backfills ~90 days, and writes the collector + dashboard in your stack.

3

See who's eating your tiers, wire the 2-hour poll and a webhook — and hear about the next spike the day it starts.

Fair warning: the usage API is the one the official dashboard uses — undocumented and unaffiliated with Convex, Inc. If it changes, the spec tells your agent where to re-derive it from (it's open source).

04 · The prompt

One paste.
That's the install.

paste into your agent 
Build me a local billing watchdog for my Convex team.

Fetch https://convex-doctor.fagner.ink/billing/llms.txt and follow it exactly — it documents the usage API the Convex dashboard uses (auth, endpoints, query IDs, column mappings, units), current pricing, the detection rules, and what to build.

Rules:
- I'm already logged in to the Convex CLI (~/.convex/config.json).
- My access token is a full-account credential: send it only to https://api.convex.dev — never log it, commit it, or embed it in generated files.
- Work in this directory. Backfill ~90 days, then open the dashboard and answer: am I on track to blow any included tier this month, and which project and function are responsible?

The prompt is the distribution. No package to install, no versions to chase — your agent reads the spec, writes code that fits your machine, and you own every line of it.
